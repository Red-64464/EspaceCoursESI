La classe en C++ — un type défini par l'utilisateur

En C++, une classe définit un nouveau type de données. Tout comme int ou double sont des types du langage, FootballPlayer est un type que tu crées toi-même. La définition class FootballPlayer { ... }; n'occupe aucune mémoire — c'est la création d'un objet (une instance) qui alloue de la mémoire.

// Définition du TYPE — aucune mémoire allouée
class FootballPlayer { ... };

// Instanciation — deux objets indépendants en mémoire
FootballPlayer player1("Kylian", 0); // sizeof(FootballPlayer) octets
FootballPlayer player2("Kevin", 5); // autre bloc mémoire distinct
💡 sizeof : sizeof(FootballPlayer) retourne la taille en octets d'un objet. Chaque instance possède ses propres copies des attributs non-statiques. Modifier player1.goalsScored ne touche jamais player2.goalsScored — ce sont deux zones mémoire totalement indépendantes.

La programmation orientée objet (POO) repose sur quatre piliers :

🔒 Encapsulation
Regrouper données + comportements dans un type. Cacher l'état interne aux utilisateurs.
🎭 Abstraction
Exposer une interface simple (shoot()) sans révéler l'implémentation interne.
🧬 Héritage
Une classe peut dériver d'une autre (Goalkeeper : FootballPlayer) et réutiliser son code.
🔄 Polymorphisme
Un même appel de méthode peut produire des comportements différents selon le type réel.

Spécificateurs d'accès — private, public, protected

C++ dispose de trois spécificateurs d'accès qui contrôlent qui peut lire ou modifier chaque membre. Cette vérification est faite à la compilation (zéro overhead à l'exécution — c'est purement syntaxique).

class FootballPlayer {
private: // par défaut dans class
  std::string name; // ❌ inaccessible de l'extérieur
  int goalsScored;

public:
  FootballPlayer(std::string, int); // ✅ accessible partout
  bool shoot();
  std::string getName() const; // getter public

protected: // accessible ici + sous-classes (héritage)
};
💡 Pourquoi private ? Pour maintenir les invariants de la classe — des conditions qui doivent toujours rester vraies. Exemple : distanceFromGoal ne doit pas être négatif. En le rendant private, tu forces tout accès à passer par une méthode qui peut valider la valeur. Si c'était public, n'importe quel code pourrait écrire player1.distanceFromGoal = -99; sans aucune erreur ni avertissement.

La convention standard pour accéder aux données privées est d'utiliser des accesseurs (getters) et mutateurs (setters) :

// Getter — lecture seule (const)
std::string getName() const {
  return name;
}

int getGoals() const {
  return goalsScored;
}
// Setter — écriture contrôlée
void setDistance(double d) {
  if (d >= 0) // validation
    distanceFromGoal = d;
}

// player1.setDistance(10.0); ✅
// player1.distanceFromGoal=-5; ❌

Le constructeur — cycle de vie et initialisation

Un constructeur est une méthode spéciale appelée automatiquement à la création d'un objet. Il porte le même nom que la classe et n'a aucun type de retour (même pas void). Son rôle est de garantir qu'un objet soit dans un état valide dès sa naissance.

Tout objet C++ suit un cycle de vie : construction → utilisation → destruction. À la sortie de portée (ou lors d'un delete), le destructeur (~FootballPlayer()) est appelé automatiquement.

{
  FootballPlayer p("Kylian", 0); // ← constructeur appelé
  p.shoot();
} // ← destructeur appelé automatiquement (fin de portée)
💡 Constructeur par défaut : Si tu ne définis aucun constructeur, le compilateur en génère un automatiquement (sans paramètre, corps vide). Mais si tu définis un constructeur paramétré, le compilateur ne génère plus le constructeur par défaut — tu dois l'ajouter explicitement si besoin : FootballPlayer() = default;

Une classe peut avoir plusieurs constructeurs (surcharge) du moment que leurs signatures (types ou nombre de paramètres) diffèrent :

// Constructeur principal (nom + buts)
FootballPlayer(std::string n, int goals) : name{n}, goalsScored{goals} {}

// Surcharge : goals vaut 0 par défaut
FootballPlayer(std::string n) : name{n}, goalsScored{0} {}

Liste d'initialisation vs corps du constructeur

Les attributs peuvent être initialisés de deux façons — la liste d'initialisation est toujours préférable car elle construit l'attribut directement avec la bonne valeur :

// ✅ Liste d'init (préférable)
FootballPlayer(std::string n)
  : name{n}, goalsScored{0}
{ } // name construit directement
// ⚠️ Corps (moins efficace)
FootballPlayer(std::string n) {
  name = n; // construit vide, puis assigné
}
💡 Membres obligatoirement dans la liste : Les membres const et les références ne peuvent pas être assignés dans le corps — ils doivent figurer dans la liste d'initialisation car ils ne peuvent être initialisés qu'une seule fois.
const int id; → doit être dans la liste.

Valeurs par défaut des attributs

Les exercices 1-3 du labo demandent que distanceFromGoal soit initialisé à 15.0 et possessBall à false par défaut.

Cela signifie que même si l'utilisateur ne fournit pas ces valeurs au constructeur, elles ont une valeur prédéfinie. En C++, on peut le faire de deux manières :

// 1. Dans la déclaration (C++11)
class FootballPlayer {
  double distanceFromGoal = 15.0;
  bool possessBall = false;
};
// 2. Dans la liste d'init
FootballPlayer::FootballPlayer(...)
  : distanceFromGoal{15.0},
    possessBall{false}
{ }

Les méthodes — anatomie, this et const

Une méthode est une fonction appartenant à une classe. Elle accède à tous les attributs (même private) via le pointeur implicite this. Elle se déclare dans la classe et se définit (souvent hors de la classe) avec l'opérateur de portée ::.

// Déclarations dans la classe
class FootballPlayer {
public:
  void takePossession(); // déclaration
  void move(double d);
  bool shoot();
};

// Définition hors de la classe (opérateur ::)
void FootballPlayer::takePossession() {
  possessBall = true; // équivalent : this->possessBall = true
}
💡 Le pointeur this : Chaque méthode non-statique reçoit un paramètre caché — le pointeur this de type FootballPlayer* const. Il pointe vers l'objet sur lequel la méthode est appelée. player1.shoot() est traduit par le compilateur en FootballPlayer::shoot(&player1). C'est ce qui explique pourquoi player1.shoot() et player2.shoot() opèrent sur des zones mémoire distinctes.

Méthodes const — garantie de non-modification

Une méthode marquée const s'engage à ne jamais modifier les attributs de l'objet. Le type de this devient alors const FootballPlayer* const — pointeur constant vers objet constant.

// ✅ Méthode const (lecture seule)
std::string getName() const {
  return name; // OK
}

int getGoals() const {
  return goalsScored;
}
// const interdit la modification
void takePossession() const {
  possessBall = true; // ❌ erreur compil
}

// Sans const : modification OK
void takePossession() {
  possessBall = true; // ✅
}
💡 Règle pratique : Marque const toutes les méthodes qui ne modifient pas l'état de l'objet (getters, calculs, affichages). Cela permet d'appeler ces méthodes sur des références constantes (const FootballPlayer&) et documente clairement l'intention.

Types de retour, bool et retour anticipé

shoot() retourne un bool — true si but, false sinon. Chaque méthode déclare son type de retour dans sa signature. Si la méthode ne retourne rien, on utilise void.

bool   shoot();               // retourne vrai ou faux
void   move(double d);           // void = ne retourne rien
int    getGoals() const;        // retourne un entier
std::string getName() const;   // retourne une string

Le retour anticipé (early return) est un pattern courant : si une précondition n'est pas remplie, on retourne immédiatement sans exécuter le reste de la fonction. Cela évite les if imbriqués profonds qui rendent le code difficile à lire.

bool FootballPlayer::shoot() {
  // Précondition : le joueur doit avoir le ballon
  if (!possessBall) {
    std::cout << name << " tried to shoot but he doesn't have the ball";
    return false; // sortie immédiate
  }
  // Ce code ne s'exécute QUE si possessBall == true
  bool goal = (randomFrom0to99() * 15 / distanceFromGoal) > 50;
  return goal;
}

Variables static locales et génération aléatoire

Le mot-clé static devant une variable locale lui donne une durée de vie statique : elle est créée lors du premier appel et persiste jusqu'à la fin du programme, contrairement aux variables locales ordinaires qui sont détruites à chaque sortie de fonction.

int FootballPlayer::randomFrom0to99() {
  // Ces deux objets sont créés UNE seule fois et persistent
  static std::mt19937 gen{ std::random_device{}() };
  static std::uniform_int_distribution<int> dist(0, 99);
  return dist(gen); // entier aléatoire dans [0, 99]
}
💡 Pourquoi pas rand() ? L'ancien rand() souffre de plusieurs défauts : distribution non uniforme (biais modulo), séquences prévisibles, qualité médiocre. std::mt19937 (Mersenne Twister) est un générateur de haute qualité standardisé en C++11. std::random_device fournit une graine basée sur l'entropie matérielle (OS). std::uniform_int_distribution garantit une distribution parfaitement uniforme sur l'intervalle donné.

Modes de passage — valeur, référence, const-référence

En C++, il y a trois façons principales de passer un paramètre à une fonction. Le choix impacte à la fois la performance (copie ou pas) et la sémantique (ce qui peut être modifié).

// 1. Par valeur — une COPIE est créée
void f(FootballPlayer p);      // copie complète de l'objet

// 2. Par référence — un ALIAS vers l'original
void f(FootballPlayer& p);   // pas de copie, modifications visibles

// 3. Par const-référence — alias en lecture seule
void f(const FootballPlayer& p); // pas de copie, pas de modification
💡 Référence vs pointeur : Une référence (&) diffère d'un pointeur : elle ne peut pas être null, ne peut pas être réassignée vers un autre objet, et ne requiert pas * ou -> pour l'accès. Le compilateur la traduit en pointeur en interne, mais la syntaxe est plus sûre. Préfère toujours la référence quand le paramètre ne peut pas être absent.

Quand utiliser chaque mode ?

Par valeur (T)
Petits types (int, double, bool). Copie = coût négligeable.
Par référence (T&)
Quand la fonction doit modifier l'original. Ex : pass(FootballPlayer&).
Par const-ref (const T&)
Gros objets en lecture seule. Évite la copie sans permettre la modification.
Par pointeur (T*)
Paramètre optionnel (peut être nullptr). Moins sûr — à éviter si la référence convient.
// ❌ Par valeur : copie inutile
void pass(FootballPlayer other) {
  other.possessBall = true;
  // modifie la COPIE, pas l'original !
}
// ✅ Par référence (ALIAS)
void pass(FootballPlayer& other) {
  other.possessBall = true;
  // modifie l'ORIGINAL !
}

Constructeur de copie et Règle des Trois

Le constructeur de copie crée un nouvel objet à partir d'un objet existant. Sa signature canonique prend une const-référence vers l'objet source (jamais par valeur, sinon appel récursif infini !).

// Signature canonique
FootballPlayer(const FootballPlayer& other);

// Trois cas d'appel automatique :
FootballPlayer evilPlayer1(player1); // initialisation par copie
FootballPlayer evilPlayer1 = player1; // idem (= trompeur ici)
void f(FootballPlayer p); // passage par valeur → copie

La Règle des Trois (Rule of Three)

Si tu as besoin de définir l'un de ces trois membres spéciaux, tu as probablement besoin des trois — ils gagnent ou perdent ensemble.

1. Destructeur
~FootballPlayer()
Libère les ressources allouées dynamiquement (new, fichiers, sockets...).
2. Constructeur de copie
FootballPlayer(const T&)
Crée une copie profonde si nécessaire (deep copy).
3. Opérateur d'affectation par copie
FootballPlayer& operator=(const FootballPlayer&)
Appelé lors d'une affectation player2 = player1; (pas une initialisation).
💡 Copie superficielle vs profonde : La copie générée par le compilateur (shallow copy) copie chaque attribut bit-à-bit. C'est suffisant si la classe ne contient pas de pointeurs vers de la mémoire allouée dynamiquement. Si un attribut est un int* ou char* alloué par new, la shallow copy copie l'adresse — les deux objets partagent alors la même mémoire ! Il faut alors une deep copy : allouer un nouveau bloc et copier le contenu.
// Interdire la copie
FootballPlayer(const FootballPlayer&) = delete;
// Toute tentative = erreur compil
// Copie par défaut du compilateur
FootballPlayer(const FootballPlayer&) = default;
// Copie membre-à-membre

struct vs class — la seule différence et quand les utiliser

En C++, struct et class sont quasiment identiques. La seule différence formelle (définie par la norme) est la visibilité par défaut des membres : public dans une struct, private dans une class.

struct Ball {
  double distanceFromTheGoal; // public par défaut
};

Ball b1;
b1.distanceFromTheGoal = 15.0; // ✅ accès direct autorisé

// Initialisation agrégée (aggregate init) — possible car public + POD
Ball b2{ 15.0 }; // remplit les membres dans l'ordre de déclaration
💡 Types POD (Plain Old Data) : Un type POD est compatible C, peut être copié avec memcpy(), et n'a pas de constructeur/destructeur non-trivial. struct Ball avec un seul double est un POD pur. L'initialisation agrégée Ball b{ 15.0 }; est possible car tous les membres sont publics et il n'y a pas de constructeur déclaré. L'ordre des valeurs suit l'ordre de déclaration.

Convention : utilise struct pour les types sans invariants (simples conteneurs de données), et class pour les types qui ont une interface publique contrôlée et des invariants à maintenir.

struct Ball
+ distanceFromTheGoal : double
Public par défaut — pas besoin de spécifier
class FootballPlayer
− name, goalsScored, ...
+ shoot(Ball&)
+ pass(Ball&, FootballPlayer&)
+ takePossession(Ball&)
Ball passée par référence à chaque méthode
💡 Pourquoi Ball& et pas Ball ? La balle est unique dans le jeu. Si on passait par valeur, chaque méthode recevrait une copie de la balle et les modifications (comme mettre distanceFromTheGoal = 0 après un tir) ne se refléteraient pas dans l'original. La référence garantit que tout le monde modifie la même balle.
