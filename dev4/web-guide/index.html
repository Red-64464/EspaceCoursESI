C'est quoi un son ?

Vibration · Onde · Représentation numérique

🌊 Le son = une vibration dans l'air. L'air bouge, ça crée une onde de pression qui voyage jusqu'à ton oreille → tu entends. En informatique, l'ordi ne comprend pas les vagues → il les transforme en nombres.

🌍 Dans la vraie vie

Le son est une vibration physique. L'air bouge d'avant en arrière, comme une vague dans la mer. Ton oreille capte ces vibrations et ton cerveau les interprète comme du son.

Air qui bouge → oreille → tu entends 🎵

💻 En informatique

L'ordi ne comprend pas les vagues — il comprend des nombres. Chaque nombre représente la position de la vibration à un instant précis. Plein de nombres mis bout à bout = le son numérique.

Vague 🎵 → nombres 💻 0.0, 0.4, 0.8, 1.0, 0.7, 0.4, 0.0, -0.4, -0.8, -1.0, ...

Samples & Buffer

Échantillon · Tableau de valeurs

📸 Un sample = une mesure du son à un instant précis. Comme une photo d'une vidéo. Un son complet = plein de photos (samples) mises à la suite. Tous ces samples ensemble forment un buffer (un tableau de nombres).

📹 Analogie vidéo

Vidéo = plein d'images
Son = plein de nombres

Plus il y a d'images → plus la vidéo est fluide. Plus il y a de samples → plus le son est précis.

🔢 En pratique

Temps → t1 t2 t3 t4 t5 Valeurs → 0.0 -0.1 -0.5 -0.7 ...

Chaque valeur = un sample. L'ensemble = un buffer.

06

Buffer circulaire & Delay

Écho · Mémoire du passé · Read / Write

🎯 Le but : créer un écho. BOUM → boum → boum. Pour ça, il faut se souvenir du passé. L'ordi oublie tout par défaut → on crée une mémoire = buffer circulaire.

⏱️ Le delay (écho)

On rejoue le même son plus tard. Le son actuel est mélangé avec un ancien son stocké dans le buffer.

sortie = son actuel + ancien son (buffer)

🔄 Pourquoi "circulaire" ?

Le son ne s'arrête jamais, mais la mémoire est limitée. Quand on arrive à la fin du buffer → on revient au début. Comme un tapis roulant.

🔄 Buffer circulaire interactif — WRITE & READ

WRITE = j'écris le son actuel (présent)  |  READ = je lis un ancien son (passé)  |  distance = delay

✍ WRITE 📖 READ
3
Clique "Avancer 1 pas" pour voir le buffer tourner.

Ce qu'on fait à CHAQUE instant

1 Je prends le son actuel
2 Je lis un ancien son (dans le buffer)
3 Je mélange les deux
4 J'écris le nouveau dans le buffer
🎡 Analogie : Imagine un tapis roulant. Tu poses un son dessus. Il tourne. Il revient plus tard. Quand il revient → tu l'entends avec le nouveau son. = écho !

Le Séquenceur

Grille · BPM · Step · Drum Machine

🥁 C'est quoi une boîte à rythmes ? Une machine qui joue des sons automatiquement et en boucle. Un séquenceur gère une grille : lignes = instruments, colonnes = pas de temps.
Temps → [ 1 ][ 2 ][ 3 ][ 4 ][ 5 ][ 6 ][ 7 ][ 8 ]... Kick → [ X ][ ][ X ][ ][ X ][ ][ ][ ] Snare → [ ][ X ][ ][ X ][ ][ X ][ ][ ] Hi-Hat→ [ X ][ X ][ X ][ X ][ X ][ X ][ X ][ X ]

Un curseur avance case par case → case active ✅ = son joué → arrivée à la fin → retour au début 🔁

🥁 Séquenceur interactif 4 pistes × 16 steps

Clique sur les cases pour les activer/désactiver, puis appuie sur Play !

120 BPM
● Kick ● Snare ● Hi-Hat ● Clap

Calcul : BPM → durée d'un step

durée_beat = 60 / BPM // en secondes durée_step = durée_beat / 4 // 4 subdivisions par beat // À 120 BPM : durée_beat = 60 / 120 = 0.5 s durée_step = 0.5 / 4 = 0.125 s = 125 ms
💡 120 BPM → 1 beat = 500ms → 1 step = 125ms

Pourquoi 256 frames ?

📦 Le programme ne traite PAS 44100 frames d'un coup. Il travaille par petits blocs de 256 frames (= 1 buffer). Il envoie ~172 buffers par seconde (44100 / 256 ≈ 172). Toutes les ~6 ms, le programme est appelé pour fournir le buffer suivant.

🔑 Point clé : la case allumée ne joue PAS "juste 256 frames"

Elle déclenche la lecture complète du son. Le son est ensuite lu petit morceau par petit morceau (buffer après buffer) jusqu'à ce qu'il soit terminé. La ligne jaune = bouton "START". Le player = continue la lecture jusqu'à la fin du WAV.

📁 Étape 1 — Le fichier WAV sur le disque

💬 En mots simples : Sur ton disque dur, tu as des fichiers comme kick.wav, snare.wav, etc. Ce sont juste des fichiers — des suites d'octets (0 et 1) stockés sur le disque. Le programme ne peut PAS jouer directement un fichier. Il faut d'abord le lire et le convertir en nombres qu'il comprend.

📂 C'est quoi un .wav ?

Un fichier WAV c'est comme une boîte avec deux compartiments :

1. Le header (l'étiquette) → dit "mon son est à 44100 Hz, en stéréo, en 16 bits"

2. Les données (le contenu) → les vrais échantillons du son, stockés en octets bruts

🎯 Dans ton projet

Tu as 4 fichiers WAV :

kick.wav → le son de la grosse caisse 🥁 snare.wav → la caisse claire hihat.wav → le charleston 🎩 clap.wav → le clap 👏

Chaque fichier = un instrument différent.

💡 Point important : Le fichier WAV contient des octets bruts — pas des nombres entre -1 et +1. Il faut une conversion pour que le programme puisse les utiliser. C'est l'étape suivante !

📥 Étape 2 — Le chargement en mémoire

💬 En mots simples : Quand tu choisis un fichier WAV dans l'interface, le programme fait 3 choses : il lit le fichier, il convertit les données en nombres compréhensibles (des float entre -1 et +1), et il range tout ça dans un grand tableau en mémoire. Ce tableau s'appelle le sample buffer.
📥 Les 3 étapes du chargement
1

📖 Lire les octets bruts

La fonction SDL_LoadWAV() ouvre le fichier et lit tous les octets dans la RAM. Elle récupère aussi le header (sample rate, nombre de canaux, bit depth).

SDL_LoadWAV("kick.wav", &spec, &rawBuf, &len) Résultat : rawBuf = [01, 3F, A2, B7, C4, 00, FF, ...] spec = { freq: 44100, channels: 2, format: S16 }
2

🔄 Convertir en float stéréo 44100 Hz

Le fichier WAV original peut être en n'importe quel format (8 bits, 16 bits, mono, 22050 Hz...). La fonction SDL_ConvertAudioSamples() convertit tout vers le format du projet :

Float 32 bits Stéréo (2 canaux) 44100 Hz

Si le fichier est mono → il le duplique en stéréo.
Si le fichier est en 16 bits → il convertit en floats (-1.0 à +1.0).
Si le sample rate est différent → il rééchantillonne.

🎼 Étape 3 — Le Step Sequencer (chef d'orchestre)

💬 En mots simples : Le séquenceur, c'est comme un métronome intelligent. Il ne joue AUCUN son lui-même. Son seul travail : regarder sa montre, et quand c'est le bon moment, il dit aux players "COMMENCE !". C'est le chef d'orchestre — il donne les ordres, mais ce sont les musiciens (les players) qui jouent.

🎯 Ce que le séquenceur fait

✓ Il compte le temps qui passe
✓ Il sait quel step on est (0 à 15)
✓ Il regarde la grille : "est-ce que cette case est allumée ?"
✓ Si oui → il dit au player : start()

🚫 Ce que le séquenceur ne fait PAS

✗ Il ne lit AUCUN son
✗ Il ne touche AUCUN buffer
✗ Il ne mélange rien
✗ Il ne sait même pas ce qu'est un fichier WAV

Il est 100% timing, 0% audio.

Comment il compte le temps

// À chaque callback (~6ms), il ajoute un petit bout de temps : bufferDuration = 256 / 44100 = 0.0058 secondes // durée d'un buffer durationInBeats = (0.0058 × BPM) / 60 // converti en "beats" beatAccumulator += durationInBeats // on accumule // Quand l'accumulateur dépasse 0.25 (= 1 step) : if (beatAccumulator >= 0.25) { beatAccumulator -= 0.25; currentStep = (currentStep + 1) % 16; // step suivant (0→15→0) // Pour chaque piste : si la case est allumée → START ! for (track = 0; track < 4; track++) { if (grille[track][currentStep] == true) { players[track].start(); // "Joue ton son !" } } }

Exemple concret à 120 BPM

// À 120 BPM : durationInBeats = (0.0058 × 120) / 60 = 0.01161 beats par callback // Il faut ~21.5 callbacks pour atteindre 0.25 : 0.25 / 0.01161 ≈ 21.5 callbacks → soit ~125 ms par step // 16 steps × 125 ms = 2 secondes pour une mesure complète 🔁

C'est quoi "déclencher un son" ?

🔑 Déclencher un son = appeler player.start(). C'est tout. Ça veut dire : "remets ta position de lecture au début (ou à la fin si reverse) et mets-toi en mode playing = true". Le player commencera à lire son sample buffer au prochain callback. Le séquenceur n'attend PAS que le son finisse — il donne l'ordre et passe au step suivant.
// "Déclencher un son" en vrai, c'est : player.start(reverse): position = reverse ? (totalFrames - 1) : 0 // début ou fin playing = true // "je suis en train de jouer" // C'est TOUT. Pas de lecture, pas de son. Juste un flag.

🎵 Étape 4 — L'Audio Player (le musicien)

💬 En mots simples : L'Audio Player, c'est le musicien. Il a son instrument (le sample buffer) et quand le chef dit "joue !", il commence à lire les notes une par une. Mais attention : il ne lit pas tout d'un coup ! Il lit juste un petit bout à chaque fois (256 frames), et il continue au callback suivant. Comme quelqu'un qui lit un livre page par page, pas tout le livre en une fois.
⚠ LE piège n°1 à éviter : "Le player joue le son en entier quand le séquenceur dit START".
NON ! Le player ne lit que 256 frames à la fois (= 1 callback = ~6ms de son). Le son complet est étalé sur plein de callbacks successifs. Un kick qui dure 0.5 seconde = ~86 callbacks pour le lire en entier !
📖 Lecture d'un son Le player lit morceau par morceau

Que se passe-t-il À CHAQUE callback ?

1

Vérifier : "est-ce que je suis en train de jouer ?"

Si playing == false → écrire du silence (que des 0.0) dans le track buffer et s'arrêter là.

2

Lire 256 frames depuis le sample buffer

Pour chaque frame (de 0 à 255), lire les 2 floats (gauche + droite) depuis le sample buffer à la position actuelle, les multiplier par le volume, et les écrire dans le track buffer.

// Pour chaque frame i de 0 à 255 : trackBuffer[i × 2] = sampleBuffer[position × 2] × volume // gauche trackBuffer[i × 2 + 1] = sampleBuffer[position × 2 + 1] × volume // droite position++ // avance !
3

Vérifier : "est-ce que j'ai fini le son ?"

Si position >= totalFrames → mettre playing = false. Au prochain callback, le player écrira du silence.

🎮 Simulateur de lecture Vois le player avancer dans le sample buffer

Chaque clic = 1 callback (~6ms). Le player lit 256 frames du sample buffer et avance sa position.

Frame 0 Frame 2000
position : 0 / 2000 — playing : true — callback : #0
Clique pour avancer le player de 256 frames.

🔄 Comment fonctionne le reverse ?

💬 C'est super simple : au lieu de commencer au début et d'avancer (position++), le player commence à la fin et recule (position--). C'est tout. Le son est lu à l'envers !

▶ Normal (forward)

position = 0 // commence au début // à chaque frame : position++ // avance // fin quand position >= totalFrames

Le son joue du début à la fin → 🥁

◀ Reverse

position = totalFrames - 1 // commence à la fin // à chaque frame : position-- // recule // fin quand position < 0

Le son joue de la fin au début → 🥁🔙

⏱ Étape 5 — Le Delay (l'écho)

💬 En mots simples : Le delay, c'est comme un magnétophone à bande. Imagine une bande magnétique qui tourne en boucle. Tu enregistres dessus (WRITE), et un peu plus loin sur la bande, une tête de lecture relit ce que tu as enregistré avant (READ). Le résultat : tu entends le son d'avant mélangé avec le son d'maintenant → écho !
⚠ Attention : Le delay modifie directement le track buffer. Il ne crée pas un nouveau buffer. Après le delay, le track buffer contient le son original + l'écho mélangé.

Le buffer circulaire du delay

📏 Taille

88200 floats = 44100 × 2 canaux = 1 seconde de son stéréo en mémoire. C'est la durée maximale de l'écho.

🔄 Circulaire ?

Quand on arrive au bout du buffer (position 88199), on revient à 0. C'est un cercle — il ne se remplit jamais, il écrase les anciennes valeurs.

Ce qui se passe pour chaque sample

// Pour chaque sample du track buffer (512 au total) : // 1. Calculer où lire dans le passé delaySamples = delayTime × 44100 × 2 // ex: 0.3s → 26460 floats readIndex = (writeIndex - delaySamples) % 88200 // position "dans le passé" // 2. Lire l'ancien son (l'écho) delayed = circularBuffer[readIndex] // le son d'il y a 0.3s // 3. Mélanger : son actuel + écho trackBuffer[i] = trackBuffer[i] + delayMix × delayed // ↑ son actuel ↑ force de l'écho (0.0 à 1.0) // 4. Écrire le résultat dans le buffer circulaire (pour le futur !) circularBuffer[writeIndex] = trackBuffer[i] // ← FEEDBACK ! // 5. Avancer les pointeurs readIndex = (readIndex + 1) % 88200 writeIndex = (writeIndex + 1) % 88200

🔁 Le feedback — pourquoi l'écho se répète

💬 Le secret : À l'étape 4, on écrit dans le buffer circulaire la valeur qui contient déjà l'écho (trackBuffer[i] = original + écho). Donc quand cette valeur sera relue plus tard, elle créera un écho de l'écho. Et ainsi de suite → l'écho se répète et s'atténue à chaque fois.
// Avec delayMix = 0.5 et delayTime = 0.3s : Son original : BOUM (amplitude 1.0) 1er écho (après 0.3s) : boum (amplitude 0.5) // 1.0 × 0.5 2ème écho (après 0.6s): boum (amplitude 0.25) // 0.5 × 0.5 3ème écho (après 0.9s): boum (amplitude 0.125) // 0.25 × 0.5 ... et ça continue en s'atténuant jusqu'au silence
💡 delayMix contrôle la force de l'écho :
0.0 = pas d'écho (le delay ne fait rien)
0.3 = écho léger
0.5 = écho moyen
0.8 = écho fort (se répète longtemps)
≥ 1.0 = ⚠ DANGER ! L'écho ne s'atténue JAMAIS → le son explose !

🎛 Étape 6 — Le Mixer (le mélange)

💬 En mots simples : Le mixer, c'est la table de mixage. Il prend ce que chaque musicien joue en ce moment (les 4 track buffers) et il les additionne pour faire un seul son. C'est vraiment juste une addition.
⚠ LE piège n°2 : "Le mixer mélange tous les buffers de la seconde".
NON ! À un instant T, chaque piste a UN SEUL petit buffer de 256 frames (le track buffer de CE callback). Le mixer additionne seulement ces 4 petits buffers. Pas 172. Pas tout le fichier. Juste 4.

// C'est littéralement 3 lignes : for (i = 0; i < 512; i++) { output[i] = track1[i] + track2[i] + track3[i] + track4[i]; } // C'est tout. Pas de magie. Juste une addition.

❌ FAUX — le mixer mélange 172 buffers

Non ! Il n'existe qu'un seul buffer actif par piste à chaque instant. Le mixer en reçoit 4 (un par piste) et les additionne.

✅ VRAI — le mixer additionne 4 buffers

À chaque callback, il prend les 4 track buffers (après le delay) et fait : output = T1 + T2 + T3 + T4, sample par sample.

🔊 Étape 7 — Audio Out (la sortie)

💬 En mots simples : L'output buffer (le mélange final) est donné à PortAudio. PortAudio le passe à la carte son de ton ordi. La carte son convertit les nombres en signal électrique. Le signal passe dans les haut-parleurs ou le casque. Les haut-parleurs bougent → l'air bouge → ton oreille capte la vibration → tu entends le son ! 🎧

🔄 Un callback complet — que se passe-t-il en 6ms ?

Le vrai code du callback

int audioCallback(float* output, int frames) { // frames = 256, output = le buffer que PortAudio attend // 1. Le séquenceur vérifie le timing et déclenche les sons sequencer.process(drumMachine, players); // 2+3. Pour CHAQUE piste : float trackBuffers[4][512]; // 4 buffers temporaires (1 par piste) for (i = 0; i < 4; i++) { // 2. Le player lit 256 frames de son sample buffer players[i].process(trackBuffers[i], frames, volume[i], muted[i]); // 3. Le delay modifie le track buffer (ajoute l'écho) delays[i].process(trackBuffers[i], frames, delayTime, delayMix, enabled); } // 4. Le mixer additionne les 4 track buffers mixer.process(output, trackBuffers[0], trackBuffers[1], trackBuffers[2], trackBuffers[3], frames); // 5. output est automatiquement envoyé à la carte son par PortAudio return paContinue; // "continue à m'appeler !" }
💡 Les 4 track buffers sont des variables locales (sur la stack). Ils sont créés au début du callback et détruits à la fin. Ils n'existent que pendant ~6ms. Au callback suivant, on en crée 4 nouveaux.

⚠ Erreurs et malentendus fréquents

🚨 Les erreurs les plus courantes des étudiants. Lis-les toutes — tu en as probablement cru au moins une !
❌ "Le séquenceur joue les sons"
✅ Le séquenceur ne joue RIEN. Il dit juste aux players "commence !". C'est un chef d'orchestre, pas un musicien. Il ne touche aucun sample, aucun buffer audio.
❌ "Le son sort d'un coup quand la case est allumée"
✅ La case allumée déclenche le son (= appelle start()). Mais le son est lu petit bout par petit bout : 256 frames par callback, sur des dizaines de callbacks successifs. Un kick de 0.5s = ~86 callbacks.
❌ "Le mixer mélange tous les buffers de la seconde (172 buffers)"
✅ Le mixer ne reçoit que 4 buffers à chaque callback (1 par piste). Il les additionne. C'est tout. Il n'a aucune mémoire des callbacks précédents.
❌ "Le sample buffer et le track buffer c'est la même chose"
✅ Non ! Le sample buffer = le son COMPLET en mémoire (des milliers de frames). Le track buffer = un PETIT BOUT de 256 frames lu pendant 1 callback. C'est comme "le livre entier" vs "la page qu'on lit maintenant".
❌ "Le delay crée un nouveau buffer"
✅ Le delay modifie directement le track buffer. Après le delay, le track buffer contient le son original + l'écho mélangé. Pas de nouveau buffer créé.
❌ "Le fichier WAV est relu à chaque fois qu'on joue le son"
✅ Le fichier WAV est lu UNE SEULE FOIS (au chargement). Ensuite, le sample buffer reste en RAM. Quand on "joue", on lit depuis la RAM (ultra rapide), jamais depuis le disque.
❌ "Le reverse inverse le buffer en mémoire"
✅ Le buffer n'est PAS modifié. Le player lit simplement dans l'autre sens : position commence à la fin et décrémente au lieu d'incrémenter. Les données restent identiques.

Formules utiles

Buffers par seconde
44100 / 256 ≈ 172
→ Le programme est appelé ~172 fois par seconde pour fournir un nouveau buffer de 256 frames.
Samples stéréo par seconde
44100 × 2 = 88200
→ En stéréo, chaque frame = 2 samples (G+D). Donc 88200 valeurs pour 1 seconde.
Durée d'un beat
durée = 60 / BPM
→ À 120 BPM : 60/120 = 0.5 seconde par beat.
Durée d'un step
step = (60 / BPM) / 4
→ 4 subdivisions par beat. À 120 BPM : 0.125s = 125ms par step.
Écho (delay)
sortie = actuel + buffer[read]
→ On mélange le son présent avec un ancien son lu dans le buffer circulaire.
Index circulaire
index = (pos + 1) % taille
→ Le modulo (%) fait revenir l'index au début quand on atteint la fin du buffer.
